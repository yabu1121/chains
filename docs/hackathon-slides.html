01 · What is chains

「誰とつながっているか」を
そのまま体験にする

  • フレンドオンリー設計：相互承認した相手とだけつながる、静かな社会
  • グラフが主役：全ユーザーを node、承認済みの友達関係を link とした ネットワークを可視化
  • プログラミング言語で発見：各ユーザーの使用言語で人を探し、グラフを絞り込む
  • "chains" = 連鎖：人と人のつながりの鎖を、名前とUIで一貫して表現

Core Artifact

友達申請 → 承認 → グラフが育つ。 QR招待・申請メッセージ・アバター・プロフィール公開範囲まで、 すべて「つながり」を中心に設計。

07 · 工夫① 設計

差し替え可能な「継ぎ目」で組む

Feature-sliced

auth / user / profile / friend / network / avatar。 各機能が handler・service・repository・dto・cache を自己完結で持つモジュラモノリス。

Interface seam

store / blobstore.Store / cache.Cache / obs をインターフェース化。 本番実装とモックを同じ型で差し替え。

Graceful degradation

Redis ダウン時は in-memory cache へ。 観測（Sentry/OTel）は no-op を既定にして本番だけ差し込む。

Testability

service はモック store で純粋にユニットテスト、 統合は embedded-postgres で実DB。外部依存ゼロでフルテスト。

08 · 工夫② グラフ理論を"体験"に

ブリッジ検出 — 2つの世界をつないだ瞬間

  • 友達承認が、今まで分断されていた2クラスタを初めて結ぶ瞬間を検出
  • 定理：新しい辺がブリッジ ⟺ 両端が別の連結成分にいた
  • 承認前のグラフで BFS するだけ。届かなければブリッジ＝演出
  • 両側の人数まで出して「あなた側のN人と相手側のM人がひとつに」

アプリ名 chains を、機能で文字通り体現。

// 承認前のグラフを BFS uSide := from(u) if uSide.has(v) { return nil // 既に連結＝サイクル } // 別成分 → ブリッジ！ return &Bridge{ YourSide: len(uSide), TheirSide: len(from(v)), }

10 · 課題解決アプローチ

ぶつかった壁と、その越え方

Problem · Cookie が消える

*.run.app は別オリジン扱いで サードパーティ Cookie がブロック。→ フロントから /api を同一オリジンへ Proxy し、 認証 Cookie をファーストパーティ化。

Problem · テストに Docker

CI/ローカルで DB を立てたくない。→ embedded-postgres + miniredis で プロセス内に本物を起動。外部依存ゼロ。

Problem · キャッシュ整合

共有グラフのキャッシュが古くなる。→ versioned key（network:global:v2）+ 関係変更時に対象だけ無効化。

Problem · CIの秘密鍵

サービスアカウント鍵を置きたくない。→ Workload Identity Federation で鍵レス認証。

多言語
